# 运输计划处理

运输计划处理，即将运输订单（下称运单），通过算法优化引擎，在指定的优化参数和策略的要求下，进行优化计算，最终安排为一个个运输车次的过程，其输出将会用于实际运输的执行。 运输计划涵盖：运单维护与发起优化，查看优化请求状态，查看与调整优化车次结果，固定线路维护，优化参数维护，上传请求、承运商份额等运输计划相关的业务功能。

# 1. 运单

系统内的运单，需要指定提卸货地点，提卸货时间窗，相应的运单货物体积重量等信息。同时用户可以自定义配置相关的运单类型、运单等级等信息。用户可通过“运输计划-运单”页面新增/查看/删除运单信息，查看运单地点的地图分布，选择运单发起优化等功能。完成运输计划的第一步：选择订单发起优化。

# 1.1 运单头信息（字段说明）

  • 通用信息：
    • 运单ID：显示运单序列号信息，不可编辑，由系统自动生成。
    • 运单编号：系统运单的唯一代码，通常为上游系统的订单号码。
    • 运单跟踪编号：可自行设置运单的跟踪编号，用以标注运单的特征，便于辨别或搜索，可重复。
    • 状态：显示运单状态，不可编辑，新增时默认为“新增”，并根据优化情况由系统自动回填相应状态值，目前订单状态一共包括“新增”、“处理中”、“已分配”三种状态。
      • 新增：表示运单尚未被优化
      • 处理中：表示运单已经被选择发起优化，优化引擎正在计算处理该运单
      • 已分配：表示运单已经优化完成，被分配到了某个车次上。如果优化失败，没有将运单分配到某个车次上，则订单将返回“新增”状态
    • 所属公司/部门：设置运单的所属部门和公司信息，每一个运单必须属于某一个公司和部门进行权限控制。
    • 运单优先级：维护运单的优先级，只能输入整数，数字越小优先级越高，优化计算时，优先级高的运单在同一个车次上会安排在优先级低的运单之前。
    • 承运商/承运商简称车次号：显示运单所被分配到的承运商及车次信息，系统优化完成后自动回填，不可编辑。
    • 装车长度（米）：运单中所有商品在考虑装载时的最大长度，以此匹配车型上的最大装车长度，如果某一个运单的装车长度超过了车型上设置的最大装车长度，则这个运单不能分配到这个车型上。此装车长度为一个运单统一值，不累加，不涉及3D装载的计算。
  • 地址信息：
    • 提/卸货地信息：维护运单的提货地和卸货地，必须是系统中维护过的运输地点信息，通过放大镜选取，只能选择相同公司部门下的运输地点。提卸货地点不能相同。
    • 提/卸货时长：可以设置在提货站点的提货时长，卸货站点的卸货时长。运单上的指定提卸货时长之后，对应提卸货运输地点上的可变效率设置将不再生效，但是提卸货运输地点的上固定处理时长仍会被计算。
  • 时间窗信息：
    • 提货开始时间：到达提货点可以开始提货的最早时间。
    • 提货结束时间：到达提货点可以开始提货的最晚时间，必须晚于“提货开始时间”
    • 卸货开始时间：到达卸货点可以开始卸货的最早时间。必须晚于”提货开始时间“
    • 卸货开始时间：到达卸货点可以开始卸货的最早时间。必须晚于”卸货开始时间“

# 1.2 运单明细信息

运单明细，即为运单实际运输的单元，可以是托盘，件，箱等。可以关联实际的商品大类信息。运单明细信息为表格输入的形式，可在右上角添加新的明细行，点击行头上的“删除”可删除明细行

  • 运单明细信息：
    • 类别：设置运单明细中商品类别，默认为“件”，可在后台进行调整。
    • 数量：运单的运输单元数量，精确至小数点后两位。
    • 总重量(千克)：维护此行运单明细的总重量，单位为千克，精确至小数点后两位。
    • 总体积(立方米)：维护此行运单明细的总体积，单位为立方米，精确至小数点后两位
    • 商品代码：可下拉搜索选择或手动录入已维护的商品类别代码，选择好之后系统会自动获取商品名称，显示在“商品名称“字段下。

# 1.3 查看运单的地图分布

在运单维护页面上，可以勾选多个订单，点击“地图”按钮，系统将跳转页面，将选择的运单的提货地和目的地在地图上投射，可以按区域分色显示及按单量多少区分大小显示。方便用户快速查看运单的整体分布情况。

  • 显示提货地/显示卸货地：当打开此开关时候，地图分布上会只显示运单的提货地，或者卸货地
  • 状态：已经选择的运单，可以通过此下拉框过滤显示相应状态的运单分布情况，例如选择100个运单进行地图展示，其中10个运单是“已分配状态”。通过选择状态为“已分配”，可仅查看这10单的分布状况。
  • 地图形式：设置运单在地图上的展现形式
    • 散点图：将提货地卸货地投射在地图上，同一个地点的运单量越多，颜色越深
    • 链路图：将运单的提货地和卸货地用线连接起来，可以查看运单流向
    • 区域分布：根据运输地点所属的区域，进行区分颜色显示，不同的颜色代表不同的区域，点位越大说明在这个运输地点的单量越多。

# 1.4 选择运单进行优化

用户可通过运单页面，选择需要优化的运单，点击“优化”按钮，发起优化进行计算。目前支持两种优化方式：单击优化按钮，即按照勾选的运单进行优化。注意勾选运单时不可跨页，只会选择当前页码上勾选的订单发起优化；

运单优化时只会将”新建“状态的运单发送至优化引擎计算，已经“处理中”或者“已分配”的运单不会纳入计算范围。

  • 勾选运单优化：单击优化按钮，即按照勾选的运单进行优化。注意勾选运单时不可跨页，只会选择当前页码上勾选的订单发起优化；
  • 全选优化：鼠标悬停在”优化“按钮时，可下拉选择“全选优化”，将按照当前查询条件能查出来的运单全部发起优化，无需勾选运单

点击优化之后会弹出参数集选择的页面（参数及的设置请参考本文档后续章节的说明）。通过下拉选择合适的参数集进行优化，系统会根据这一参数设置进行后续优化，在参数集选择页面上可以查看当前发起优化请求的总运单数/站点数/总体积/总重量/总件数等相关信息。

# 2. 车次

车次为优化引擎计算的产出物，包含一个车次的体积重量装载率，开始/结束时间，车型、车牌，承运商，需要去访问的站点，每个站点的顺序，每个站点下执行的动作（装货/卸货等），站点的计划到达计划离开时间等信息。作为执行运输任务的指导。用户可通过“运输计划-车次”页面，新增/查看/删除运单信息方便调度/司机查看任务

系统内的车次功能包括查看车次、地图展示路径、调整车次、车次下载、确认车次等功能。

# 2.1 车次信息（字段说明）

  • 通用信息：

    • 车次号：系统中车次的唯一代码，由系统自动回填不可更改，主键顺序生成。
    • 描述：可人工输入车次的详细描述。
    • 跟踪编号：可自行设置车次的跟踪编号，用以标注车次的特征，便于辨别或搜索，可重复，当车次是通过固定线路模版优化出来的时，跟踪编号字段内会填入固定线路模版编号
    • 状态：显示车次状态，不可编辑，优化完成时默认为“新增”，在确认后会变为“已确认”
    • 车型代码/车型描述：自动回填车次所分配的车型，也可根据实际情况通过下拉选择的方式修改所使用的车型
    • 车牌号：自动回填车次所分配的车辆车牌号，也可根据实际情况通过查询搜索到方式修改所使用的车辆。如承运商资源内没有指定具体车辆，则此优化引擎返回时信息为空
    • 起始地代码/起始地名称/卸货地代码/卸货地名称：优化完成后自动回填车次的第一个提货地和最后一个卸货地，不可编辑。出发/回仓站点不包括在内。
    • 承运商/承运商简称：显示车次所被分配到的承运商信息，系统优化完成后自动回填，也可通过下拉搜索的方式手工调整承运商
    • 开始/结束运输时间：显示车次的开始运输时间（最早到达第一个提货站点的时间）和最后结束运输时间（离开最后站点的时间）。出发回仓时间不算在内。
    • 所属公司/部门：显示车次的所属公司和部门信息，不可编辑
    • 车次分组编号及序列号：在一车多趟的场景下，不同车次将组合为一个车次组交由一辆车执行。此时车次上会显示相同的车次组号，以及同一个组下车次的先后执行顺序

  • 汇总信息

    汇总信息内展示的是一个车次总体积/总重量/总件数/总距离/总站点数，体积利用率，重量利用率等。

  • 详细信息

    详细信息下分为3个Tab页展示站点、运单、计费的信息，优化引擎计算完成的车次会自动生成相关信息

    • 站点页：展示站点的顺序、运输地点代码/名称/地址、计划到达时间、计划离开时间、作业时间（在站点停留时长）、等待时间（到达站点之后，等待站点可以开始提卸货的时间）、站点的动作（提货/卸货/两者都有）、距离上一个站点的距离和开行时间，站点的提卸货重量/体积/件数等信息。每一行站点记录可以展开，查看这个站点下处理的运单信息。
    • 运单页：统一查看这个车次处理的所有运单信息，具体字段和运单维护页面相同。
    • 计费页：查看这个车次通过优化引擎计算出来的运费信息。一个车次可以有多条计费记录，由优化引擎生成的车次计费记录，类型为“系统生成”，且不可删除，不可调整。

# 2.2 车次地图展示与调整

用户可在车次页面，勾选车次，点击“地图与调整”按钮，查看车次在地图上的计划轨迹，并进行车次的调整。

  • 车次路径展示

    车次调整页面，单击车次列表上的车次，地图上即会显示对应车次的路径，车次列表上的不同颜色的圆点，对应地图的上的不同颜色的路径。也可在地图上点击“显示当前页”，在地图上显示车次列表上当前页的所有车次的路径。

    地图上可以选择直线/路径展示。

  • 车次调整

    在车次地图展示上，可选择车次加入待调整的列表，点击左上角的调整，将会跳转调整页面。页面上的车次将会以卡片的形式展示，可拖动整个车次卡片调整显示位置。

    目前可以进行的调整包括：

    • 单车次增加运单：点击添加运单按钮，选择运单进行添加提卸货站点已经存在在车次上，会自动合并，否则会新增相应提卸货站点
    • 单车次移除运单：至少保留一个运单在车次上。移除会更新站点的提卸货数量汇总信息
    • 单车次整体移除站点：相应运单将被从车次上移除
    • 单车次站点之间顺序的调整：可以通过拖拽站点的方式来调整

  • 多个车次之间运单/站点的交换：同样通过拖拽站点方式调整

    • 单车次调整承运商/车型：点击车型、承运商旁边的铅笔图标，进行调整

    每次调整系统都会实时保存结果，如果配置了使用计费引擎，则调整后会调用计费引擎，重新计算站点计划到达离开时间，以及相关计费信息，如果计费不通过，计划到达离开时间会被保留。但是计费信息将被清空。

    车次调整页面可选择显示站点的维度数据（体积/重量/件数），调整完毕的车次卡片上方点击X号，移出待调整列表。

# 2.3 车次添加人工计费

系统中可点击车次的详情页，点击计费信息tab查看车次的计费信息（启用计费引擎的情况下），在此页面可人工添加计费信息。人工添加的计费信息，只可选择收费项目中，类型为“人工计费”的条目。添加计费信息之后不可修改，只可删除重建。车次发生调整之后会清空所有的计费信息，重新按照系统计费计算（启用计费引擎情况下）

# 2.4 车次结果下载

在车次调整页面可以选中车次，点击“车次下载”按钮，下载车次结果。现在支持

  • 选中车次下载：将选中的车次下载为excel文件
  • 查询条件下载：将当前查询条件下的所有车次下载为excel文件

下载的excel文件包含两个sheet，一个为车次站点运单的全信息数据，另外一个为车次站点顺序的简化信息

# 2.5 车次确认

在车次调整页面可以选中车次，点击“车次确认”按钮，确认车次

车次列表页面中，系统默认只显示未确认的，状态为“新建”的车次结果；如果认可车次结果，可以在车次维护页面勾选待确认的车次进行确认。点击“确认车次”后，页面刷新，车次状态会自动由“新建”更改为“已确认”，并出现在“已确认”的车次列表中。如果需要取消已确认的车次，可以先搜索出所有“已确认”的车次，再勾选，鼠标悬浮在“确认”按钮上，会弹出“取消确认”按钮。点击“取消确认”后，页面刷新，车次状态会自动由“已确认”更改为“新建”，并出现在“新建”的车次列表中。“已确认”的车次不能点击确认，“新建”状态的车次不能取消确认

# 3. 优化请求

优化请求为用户每次调用优化引擎进行计算时的记录，用于追踪、查询、汇总每次请求的详情信息，包括请求状态与进程、提交时间、计算用时、请求失败原因、待优化的运单情况（总重量、总体积、总件数、总单数）、所使用的优化参数策略、以及生成车次结果（总车次数、总里程数）等。

在详情页中，用户可根据进一步查看每一个运单的状态以及可能的未排程原因，并确认生成的车次信息。

# 3.1 优化请求信息（字段说明）

  • 通用信息

    • 优化请求ID：显示系统自动生成的优化请求ID，系统唯一的优化请求序列号
    • 状态：系统自动回填优化请求的当前状态，包括“优化取消”、“优化失败”、“优化完成”、“优化计算中”和“待处理”共五种状态
    • 优化请求提交时间/优化开始时间/优化结束时间：系统自动回填优化时间，包括提交请求时间、开始优化时间与结束优化的时间
    • 总运单数/未分配订单数/总重量(千克)/总体积(立方米)/总距离(千米)/总件数/总车次数/计费总金额(元)：显示系统自动统计的该优化请求所涉及的各维度数据，包括待优化的运单汇总情况以及优化出的车次结果
    • 优化参数集ID/优化参数集描述：显示优化请求所选择的参数信息
    • 优化器版本：显示优化请求所选择的优化器版本
    • 距离引擎：显示优化请求所设置的距离引擎
    • 所属公司/部门：显示优化请求的所属公司和部门信息
    • API请求号：如果优化请求由API创建，API请求号显示API对应号码，如果是UI创建的，显示UI创建的请求号
    • 优化失败原因：显示优化请求提交的过程中出现失败的原因

  • 运单详情

    优化请求详情页面中分3个Tab页面分别展示该请求所涉及的运单、车次和参数策略信息。

    其中运单详情页主要展示该优化请求中所处理的运单的排程情况，如果某个运单未成功排程，系统会告知未分配的具体原因。用户可通过点击运单编号查看运单详情。

    • 运单ID/运单编号：显示该优化请求所涉及的运单ID及运单编号信息
    • 优化结果：显示某个运单最终的优化结果，分为“成功”或“失败”两种结果状态
    • 未分配原因：显示单个运单未分配到车次的具体原因，如“总重量超过可用车辆的限制”等

  • 车次详情

    车次详情页中会显示请求生成的具体的车次号，通过点击车次号可以查看车次详情

    • 车次号：车次总编号，由系统自动生成，对应车次页面中的车次号
    • 算法车次号：该请求的车次序号，由系统自动生成，取值为1，2，3，...，优化结果的车次数

  • 参数策略详情

    参数策略页中会记录优化请求所设置的参数和策略。详情可参见优化参数章节的说明。

# 3.2 优化请求状态

在优化请求汇总信息页中，可以查看和调整优化请求的状态。通过点击优化请求状态，系统会在弹窗中显示请求的进程详情。优化请求共存在“优化取消”、“优化失败”、“优化完成”、“优化计算中”和“待处理”五种状态：

  • 无论是通过前端UI操作还是API传入，发起优化后，优化请求首先进入**“待处理”状态，系统此时开始处理运单、车型、承运商资源、站点信息及距离矩阵、参数策略等必备的优化数据。处于“待处理”**状态的优化请求不能被删除；
  • 优化数据准备完成以后，算法收到开始计算，此时优化请求进入**“优化计算中”**状态
    • 如果数据完整合规、系统功能正常，算法经过计算会自动返回优化结果，系统自动更新运单状态，生成车次结果，此时优化请求状态会自动更新为**“优化完成”**；
    • 如果在数据准备或系统运算的过程中出现问题，比如当前公司部门下没有启用状态的车型、策略脚本配置错误、距离矩阵数据缺失等，优化请求状态会变更为**“优化失败”**。此时可通过“请求失败原因”检查错误根源，或联系小马驾驾工作人员确认问题；
    • 只有当优化请求处于**“优化计算中”时，才可以通过勾选请求并点击页面下方的“取消”按钮来取消当前请求。取消后，优化请求的状态变为“优化取消”**；
    • 处于**“优化计算中”状态的优化请求不能被直接删除，如果要删除需要先取消，使其变为“优化取消”**状态再进行删除操作。

# 4. 优化参数

优化参数集由优化参数和策略两部分组成，共同定义了请求时的计算逻辑。小马驾驾算法引擎基于参数列表和策略脚本，按顺序和策略语义执行优化策略，来完成将订单转化为车次的计算。用户通过设置优化参数与策略，可以便捷地处理丰富的实际调度业务场景。

在“运输计划-优化参数”列表页中，用户可以查看系统中已维护的优化参数集的基本信息，包括参数集描述、启停用状态、优化器版本、所属公司部门等，或新增/编辑/删除优化参数集数据。其中参数集描述唯一不可重复。

为了保障正常调度优化，系统中与运单相同的公司部门下应当至少维护一条处于启用状态的优化参数集。处于停用状态下的优化参数集在选择运单发起优化时不能被选择。系统中也支持同时维护多个不同公司/部门的优化参数集，但只有待优化的运单所属公司部门与所选用的优化参数集所属公司部门相一致时，方能正常发起调度优化请求。

# 4.1 优化参数

优化器版本即算法版本，可通过下拉框进行选择，每种算法有对应的参数列表和策略语法格式。当前优化器版本共有“标准版算法”和“sprit版算法”两种选项，推荐使用“标准版算法”。

优化参数详情页下方共有两个Tab页面分别呈现参数列表和策略列表信息。参数列表为键值队形式，每一个参数为一个Key，可设置对应的Value。目前可调整的优化参数包括迭代次数、最大计算时长、是否允许跨区、是否开启回仓计算、站点固定处理时长等多个参数。

# 4.2 优化策略

在优化参数详情页的第二个Tab页面中，用户可以查看或编辑优化策略信息（可能需要联系管理人员提供解锁权限的Token信息）。策略脚本的语句包括控制策略和优化策略，前者控制脚本执行的顺序、数据的选取等，后者定义优化的方向、范围、算法引擎的计算行为等。缺少策略文件信息、或策略文件中没有未注释的策略语句时，优化参数不能使用进行优化。

# 4.3 优化参数信息（字段说明）

  • 通用信息

    • 优化参数集ID：系统优化参数集的唯一代码，新建参数集时由系统自动生成，不可编辑
    • 优化参数集描述：维护优化参数集的详细描述，可自由输入相关文本信息，唯一不可重复
    • 优化器版本：即算法版本，通过下拉框选择。当前选项包括“标准版算法”和“sprit版算法”，推荐使用“标准版算法”
    • 启用/停用：标明此优化参数集是否启用，停用的优化参数集在发起优化时将无法被选择
    • 所属公司/部门：下拉选择设置优化参数的所属公司及部门信息，如只指定公司未指定部门，则该优化参数在该公司下所有部门可用，如均未指定公司及部门，则取当前账号的所属公司及部门信息
    • 创建人/创建时间/最后更新人/最后更新时间：显示优化参数集的创建和更新信息，不可编辑，系统会自动记录优化参数的创建及更新信息

  • 参数列表信息

    可根据实际业务情况设定相应参数。如果参数值留空，则算法计算时会取默认值进行计算。数值型字段若设定的参数值为0，则一般表示不考虑当前限制条件。例如最大允许跨区个数（MaxAllowCrossMaxZonesCount）设定为0时，表示不限制跨区个数（即允许无限跨区），此时与该字段参数值设置为空值效果相同。

  • 策略列表信息

    根据实际调度逻辑编写策略语句，可通过不同策略语句的嵌套模拟广泛的真实场景。

# 5. 上传请求

为了便于批量维护数据，系统开发了上传请求功能。用户可以在“运输计划-上传请求”页面中进行查看上传请求详情、删除上传请求、上传数据文件、下载上传模版等操作。

上传请求列表中展示了上传请求的基本整体情况，包括处理类型、处理数据量、错误记录数、上传请求的失败原因等信息。如果上传的文件出现了文件格式错误，比如缺少某个标签页、Mapping标签页中的值与实际不符等，则整个上传请求失败，报错信息会显示在“请求失败原因”字段中；如果上传的文件中某行数据出现错误，则报错信息会反映在上传请求详情页的上传失败明细子表中。

# 5.1 上传模版

用户可点击页面下方的“模版下载”，然后在弹窗中展开箭头找到相应的模版文件，点击模版名称，从而将模版文件下载并保存到本地。

系统目前支持的上传模板包括：

  • 运单新增

    • 运单新增的时候运输地点代码必填

    • 如果运输地点不存在，会根据模板内的数据创建运输地点

    • 如果运输地点已存在，则上传的数据将会忽略，不进行处理，不会覆盖已有数据

    • 用户可以从以下两种方式中选择来维护运单数据

      1）先通过运输地点新增模版上传运单上所有地点信息（包括提货点和卸货点），再上传运单数据；

      2）直接在运单表中提供运输地点的地址信息，一步上传运单数据及其设置的地点信息；

      如果选择方法1，则运单表中的地点地址信息列（省份、城市、地址）非必填；如果选择方法2，则运单表中的地点地址信息列必填。

    • 运单新增的时候如果提/卸货地名称为空，则系统默认取提/卸货地代码作为地点名称

    • “提/卸货时长”必须填入非负整数，单位为分钟

    • “容器类型”必须填入系统的数据字典页面中预先维护的运输单元类型（SHIP_UNIT_TYPE）数据，一般可设置为CASE（件）/PALLET（托盘）等，作为“数量”字段的单位

    • “商品大类ID”列非必填，一般如果业务场景中不涉及车型商品的对应关系时，此字段留空即可；如果填写须填入系统中已维护的商品代码

  • 运输地点新增

    • 目前“运输地点类型”分为DC（仓库）、SHIPTO（送货点）和HUB（分拨中心），无论哪种地点类型均可出现在运单的提货地和卸货地上
    • “运输地点状态”填入ACTIVE（启用）或INACTIVE（停用）
    • 经纬度信息非必填，若字段留空，则上传时由系统根据地址信息自动解析经纬度；若填入经纬度数据，则系统直接取客户提供的经纬度，不另行解析
    • “限行区域”/“业务区域”非必填，用户可手工填入地点的指定分区，也可预先在系统中维护区域围栏数据，然后由系统自动解析分配地点所属的分区

  • 运输地点可用车型新增

    • 只需维护只有部分车型可用的运输地点，如果运输地点全车型可用，则不需要维护在此表中（在全局设置中设置地点可用车型为空时可实现“全车型可用”，详见全局设置章节描述）
    • “对应类型”固定填入INCLUDE即可，表示“运输地点可用车型”对应关系，运输地点编码、车型编码需填入系统中已经维护好的运输地点代码、车型代码，未维护在表中的车型即为运输地点的不可进车型

  • 运输地点时间窗新增

    • 运输地点代码需填入系统中已维护的代码
    • “时间窗类型”包括Regular（工作时间窗）和Exception（时间窗例外），类型为Regular时“星期”字段必填，类型为Exception时“日期”字段必填
    • “开始时间”和“结束时间”的格式为“HH:MM”，取值范围为从“00:00”到“23:59”之间的任意时刻，并且需要保证结束时间晚于开始时间

  • 运输地点装卸效率新增

    • 地点代码需填入系统中已维护的运输地点代码
    • “固定处理时间”、“最大可变处理时间”、“最小可变处理时间”需输入非负数，上传后系统会经过四舍五入处理精确至小数点后两位
    • 最大可变处理时间必须大于最小可变处理时间
    • “操作类型代码”、“容器类型”、“单位时间内处理数量”、“单位时间（小时）”四个字段中如果有一个字段不为空，则剩余字段均必填
    • “处理商品代码”非必填，若留空则表示该装卸效率适用于该站点处理所有商品，若填入了具体的商品代码，则装卸效率只对某种特定的商品生效

  • 车型新增

    • 车型代码必须填入系统中已维护的代码

    • “类别”必须填入系统的数据字典页面中预先维护的车辆类型（DEVICE_TYPE）数据，一般可填入NORMAL（表示“常温车”）

    • “限行组类型”取值为0-4的正整数，分别进行如下说明：

      • 0：轿车组，表示普通小轿车

      • 1：微型货车组，主要包括金杯和依维柯两种车型

      • 2：轻型货车组，一般指载重在1.5吨以上的货车

      • 3：中型货车组，一般指9.6米以上的大型货车

      • 4：重型货车组，一般指16.5米以上的超大型货车

    • 公司部门字段非必填，如留空，则车型在上传用户所属公司层级下所有部门可用

  • 车型商品对应关系新增

    • 车型代码必须填入系统中已维护的代码
    • “对应类型”可从EXCLUDE（该车型上两种商品互斥，即不能同时装载）、RESTRICT-IN（只能装运）、RESTRICT-OUT（不能装运）中选择填入
    • 商品代码1必填，并且当“对应类型”为EXCLUDE时，商品代码2也是必填的

  • 承运商可用资源新增

    • 承运商代码、车型代码必须填入系统中已维护的代码数据
    • “车型数量”必须填入正整数
    • “车型调度成本（元）”和“距离调度成本（元/公里）”上传后系统会经过四舍五入处理精确至小数点后两位
    • “车场”必须填入系统中已维护的运输地点代码。若要实现车次回仓效果，需要在承运商资源中将车场设置为仓库代码，并在开启回仓计算优化参数

  • 固定线路新增

    • 站点代码必须填入系统中已维护的运输地点代码
    • 每一条固定线路模版至少需要包含2个站点，站点顺序必须连续并且从1开始递增排列
    • “服务开始时间”与“服务结束时间”的格式为HH:MM，且结束时间必须晚于开始时间

  • 费率新增

    • 承运商代码、公司ID、业务部门ID、起始区域代码、目的区域代码、收费项目代码、车型代码、费率阶梯代码均须填入系统中已维护的数据
    • 服务类型代码目前固定填入1即可
    • “有效期至”必须晚于“有效期起”，两个字段均精确至年月日

上传模版excel文件共有Mapping、Data和Info三个子表，其中用户只需在Data表中输入相应的数据即可，Mapping和Info表无须做任何改动，但需要保留下来，如果删除了再上传则会出错。

模板文件请遵照以下的一般格式填写：

  • 列标题为蓝色的表示该列为必填信息，若为绿色列则可选填。当然，某些上传模版的字段是否必填存在一些特例情况，请参照前文不同上传模版的详细说明；
  • “APIType”字段值参照模版文档中已给出的样例数据即可，无须修改；
  • “行类型”字段取值为H（头行）或D（明细行），如无区分固定填入H即可；
  • “公司ID”和“业务部门ID”列非必填，若留空时，系统自动取进行上传操作的用户账号的公司部门作为上传数据的公司部门信息

上传模板excel文件列标题栏右上角有红色三角形标识时，可移动鼠标至该栏的位置查看详细备注信息。

# 5.2 文件上传

批量上传文件时，用户可以进入“运输计划-上传请求”页面，点击页面下方的“文件上传”按钮，通过路径找到待上传的数据文件后点击“打开”，文件即开始上传并解析，页面随后自动刷新，显示出该上传请求的基本信息。如果上传的数据量较大，系统需要一定的时间处理上传数据，上传请求可能会停留在“处理中”状态中一段时间，此时用户可以点击“搜索”刷新页面，查看最新的上传请求处理状态。

# 5.3 上传请求详情

用户可通过点击上传请求ID查看上传请求的明细情况，包括整体的处理数量或错误数量统计，以及对每一条错误的具体提示。系统对于上传数据的报错精确到行，并会给出相应的解析失败原因，供客户复查分析以便修正。

用户可以直接在上传请求详情页面下方查看明细信息，也可以点击“下载结果文件”，在excel表格Data页的最后一列（“错误信息”列）中查看对应行数据的错误原因。

# 5.4 上传请求详情信息（字段说明）

  • 通用信息

    • 上传请求ID：系统上传请求的唯一代码，系统自动生成
    • 处理类型：上传文件所预期达到的操作类型，主要是数据（如运单/运输地点/运输地点时间窗/车型/承运商可用资源等数据）的新增；
    • 状态：标明上传请求的当前状态，共分为“处理失败”/“处理完成”/“处理中”/“待处理”四种状态
    • 处理记录数：统计上传请求所处理的数据量（行数）
    • 错误记录数：统计上传解析出错的数据量（行数）
    • 所属公司/部门：显示提交上传请求的用户的所属公司和部门信息
    • 创建人/创建时间：显示上传请求的创建信息

  • 上传请求明细信息

    • 上传失败明细ID：系统自动生成的上传失败明细唯一代码
    • 失败开始行数/失败结束行数：从用户上传文件的第二行（即除去表头部分的第一行）开始算起，用以定位上传解析失败的数据
    • 失败原因：显示数据上传解析失败的具体原因

# 6. 固定线路

固定线路模版是一条有序经过指定站点（运输地点）的车辆行驶路线，其所涉及的关键信息包括站点顺序、站点代码、计划到达时间、计划离开时间、作业时间、承运商、车型等。在特定场景中，调度人员须确保车次按照既定顺序完成运输任务，为了实现这一要求，必须在系统中预先维护固定线路模版数据，并在优化时启用按照固定线路优化的策略语句。

用户可以在“运输计划-固定线路”页面中新增/编辑/删除线路模版数据。只有处于启用的固定线路模版才会被纳入优化范围，按照模版生成车次。

# 6.1 固定线路详情

固定线路模版维护时需要指定模版名称、始发时间、站点代码、站点顺序、站点的服务开始时间、服务结束时间等内容：

  • 始发时间为到达第一个站点开始装货的时间
  • 站点代码必须填入系统中已维护的运输地点代码，或通过界面上的放大镜搜索选择运输地点
  • 站点的服务结束时间必须晚于服务开始时间
  • 每个固定线路模板至少需要包含2个站点（出发点和目的地）

预先维护了固定线路数据，并且开启固定线路策略进行优化时，系统并不一定会完全按照固定线路生成车次，因为需要综合考虑时间窗/体积重量等优化的约束条件的满足。如果车次按照固定线路生成，则固定线路模版编号会自动填入车次的跟踪编号中；如果车次的跟踪编号字段为空，则表示该车次未依照任何固定线路生成。

# 6.2 固定线路详情（字段说明）

  • 基础信息

    • 线路模版ID：系统固定线路的唯一ID，系统自动生成
      • 线路模版编号/描述：用户自定义的模版编号和备注信息
      • 是否启用：表明线路模版是否被优化请求处理
      • 所属公司/部门：显示维护固定线路数据的用户的所属公司和部门信息

  • 路线信息

    • 车型代码：表明该固定线路使用的车型代码，为空表示全车型均可使用此模版
    • 承运商代码：表明该固定线路可使用的承运商，为空表示所有承运商均可使用此模版
    • 开始时间：表明该固定线路模版的到达第一个站点的时间

  • 明细信息

    • 站点代码：该固定线路模版上的站点
    • 站点顺序：固定线路模版的站点顺序，可拖拽调整
    • 站点服务时间窗：由服务开始时间和服务结束时间组成，可以不设置，如果设置了，表明站点需要在指定的这段时间内完成服务，否则不能放入相应运单任务到该站点上

# 7. 承运商份额

实际调度排程中，针对同一个起始区域到同一个目的区域，可能有多个提供运输服务的承运商在候选池内，且报价各不相同。此时调度人员可能希望基于设定的百分比份额，将车次分配给不同的承运商，这个百分比就是承运商份额。承运商份额存在多种衡量维度，目前系统支持通过件数、车次、体积、重量4种维度的份额进行车次分配。

如果正确维护了可使用的承运商份额，并且在优化时启用承运商份额分配策略，则算法会按照份额比例调整得出总成本最小的结果。所生成的车次结果尽量贴近预先分配的比例，不需要完全一致。系统不会为了满足份额故意增加车次。

# 7.1 承运商份额详情

用户可以在“运输地点-承运商份额”页面下新增/编辑/删除承运商份额数据。其中已创建的“份额维度”不能修改保存，如果需要调整份额维度，需要删除已有条目后新建。

承运商列表页中默认显示承运商、起始区域、目的区域、份额、份额维度、有效期等基本信息，用户可以点击ID进一步查看承运商份额详情。

# 7.2 承运商份额详情（字段说明）

  • 基础信息

    • ID：系统自动生成的承运商分配份额数据唯一ID
    • 承运商代码/承运商：该份额分配记录所针对的承运商信息
    • 启用/停用：表明承运商份额数据是否被优化请求处理
    • 份额：表明承运商所需分配的份额数，最大值为100，如果希望分配80%的运输任务，则可设置为80
    • 份额维度：表明份额分配时所依据的衡量维度，当前共可从“件数”、“车次”、“体积”和“重量”中进行选择
    • 有效期起/有效期至：限定份额分配生效的时间段，非该时间段内配送的车次不考虑份额分配策略
    • 起始区域代码/起始区域/目的区域代码/目的区域：限定份额分配生效的车次起始和目的区域信息，只有车次和份额数据的起始区域或目的区域均匹配时才会考虑承运商份额分配策略
    • 所属公司/部门：显示提交上传请求的用户的所属公司和部门信息

  • 创建信息

    用户创建或编辑时承运商份额数据时，由系统自动生成不可编辑，主要包括了用户信息和更新时间。
