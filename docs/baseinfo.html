# 基础资料维护

介绍基础资料的含义、用法与维护方法。 基础资料包括包括运输地点、车型、车辆、承运商、承运商资源与商品类别。

# 1. 系统数据精度说明

  • 系统内所展现的数值类型字段的具体精度及单位说明如下表所示：
编号 字段 字段案例 精度 单位
1 长度 如长宽高等 0.01 米（M）
2 距离 0.01 公里（KM）
3 时间跨度 行驶时间 0.01 小时（H）
4 重量 0.001 公斤（KG）
5 体积 0.0001 立方米（M^3）
6 金额 0.01 元（RMB）
7 运单件数 0.01 件
8 运单托盘 0.01 托盘
9 装卸效率件数/托盘数 0（整数） 件数/托盘数
10 经纬度 0.000001
11 其他普通数量 如车辆数 0（整数）
12 时间窗 HH:MM:SS
13 百分比 0.01%

# 2. 运输地点维护

# 2.1 基础信息

维护运输地点基础信息时的规则说明：

  • 运输地点代码唯一，不允许重复。
  • 可添加站点的地址信息（输入地址并保存后，系统会自动解析该地址信息，并从距离矩阵获取经纬度信息后自动填充）。
  • 用户指定了经纬度的， 以用户指定的为准。
  • 运输地点的所属公司部门必须填写。
  • 运输地点上可维护对应所属限行区域/业务分区。
  • 限行区域可以配置相关的可用车型，例如3环内只有本地蓝牌车可用等，详见区域配置，如果运输地点配置了可用车型，则区域上的可用车型将不在启用，以运输地点的可用车型为准。
  • 具有相同业务区域的目的地才能拼载到一辆车上，为空可以和所有区域的订单拼载到一起。
  • “自动解析”开关打开时，系统根据维护的区域信息，自动解析一个运输地点属于哪个图层的区域。
  • “自动解析”开关关闭时，可以人工指定所属的区域。
  • 已经出现在运单中的运输地点不能被删除。如须删除，需要先将涉及该运输地点的所有运单删除。

# 2.2 基础信息（字段说明）

运输地点基础信息页字段说明：

  • 通用信息：
    • 运输地点代码：系统运输地点的唯一代码。
    • 运输地点描述：对运输地点的描述信息，可自由输入相关文本信息。
    • 类型：对运输地点的类型分类，运输地点可选择为：提货仓库（DC）、配送点（ShipTo）。
    • 启/停用：标志此运输地点是否启用。停用的运输地点不会纳入系统优化的范围。
    • 所属公司/部门：标明此运输地点所属的部门，不同部门下的运输地点数据是隔离的，也不可以优化到一辆车上；运输地点必须指定到具体的部门。
  • 地址信息：
    • 地址：可添加站点的地址信息。
    • 经纬度：经纬度可以在填入地址保存之后由系统自动解析获取，也可以由用户直接在页面维护。用户维护的经纬度将会覆盖系统解析的经纬度。如果留空，系统会自动解析经纬度并保存。
  • 排程要求信息：
    • 限行分区：根据国家政策或者客户需求限定的限行分区。可以配置可用车型。
    • 业务分区：根据用户需求设立的运输地点区域划分字段，用以实现特定的路线跨区限制约束。
    • 自动解析开关：开关打开时系统将会自动根据运输地点经纬度判断所属的区域。

# 2.3 时间窗

维护运输地点时间窗时的规则说明：

  • 可同时维护普通时间窗及特殊时间窗。普通时间窗以周为单位固定重复。与此同时，用户亦可以根据现实情况设置某些特定日期的时间窗，优先被优化引擎读入用以参考做排班计划。
  • 支持维护在站点提货/卸货/可提可卸三种任务类型。
  • 系统在优化时将取订单时间窗和站点时间窗的交集作为最终的时间窗约束，即排程结果将同时满足两个时间窗约束。

# 2.4 时间窗（字段说明）

运输地点时间窗页面的字段说明：

  • 工作时间窗：
    • 星期：设定一般工作时间窗信息的星期数，可从周一/周二/周三/周四/周五/周六/周日之中选择。
    • 开始时间：设定时间窗的开始时刻，精确到XX（时）:XX（分）。
    • 结束时间：设定时间窗的结束时刻，精确到XX（时）:XX（分）。星期、开始时间和结束时间共同构成整体的时间窗信息。
    • 类型：标明此时间窗信息的操作类型，可选择为卸货（DROP）、提货（PICK）、可提可卸（PICKDROP）。相同操作类型的时间窗不允许有重叠部分。
  • 时间窗例外
    • 日期：设置具体日期的时间窗（例如节假日等），在计算到具体日期时将覆盖上述普通工作时间窗。

# 2.5 装卸效率

维护运输地点的装卸效率时的规则与解释：

  • 站点的装卸效率由固定效率和可变效率组成，两者加总即为车辆在该站点的总装卸（处理）时间。
  • 固定处理时间：反映站点的固定装卸效率。
  • 可变效率：用户根据不同任务类型（装货/卸货）、容器类型（件/托盘）、商品类别（如图中的干货/冻货等）、单位时间及处理数量进行维护。系统根据这一效率针对具体的运单数量可以计算得到的基本处理时间，最后参照最大/最小可变处理时间的限制进行调整即可获得最终的可变处理时间。
  • 可变效率需维护正确的任务类型才可生效。例如送货点须维护类型为“卸货”的可变效率时，系统才能根据这一效率正确计算送货上门的可变处理时间；如果只维护了类型为“装货”的可变效率，则在该点卸货的处理时间为0。

# 2.6 装卸效率(字段说明)

运输地点装卸效率页字段说明

  • 通用效率信息：
    • 固定处理时间（小时）：可设置运输地点的固定基本装卸时间，单位为小时，精确至小数点后两位，且不能为负数。
    • 最大可变处理时间（小时）：设置运输地点的装卸时间上限，单位为小时，精确至小数点后两位，且不能为负数。
    • 最小可变处理时间（小时）：设置运输地点的装卸时间下限，单位为小时，精确至小数点后两位，且不能为负数；最大可变处理必须严格大于最小可变处理时间。
  • 可变效率信息：
    • 类型：标明该可变效率的操作类型，可从“装货/卸货”中选择，维护两种不同操作类型的可变效率。
    • 容器类型：标明运单上的商品单位，当前默认为“件”，可在数据字典页面或后台根据实际需求进行定制化配置或修改。
    • 处理的商品代码：可通过下拉搜索选择商品类别页面中已维护的商品代码，也可以直接手工输入（存在逻辑校验，只有输入已维护的商品类别代码才能正常保存）。
    • 单位时间内的处理数量：可输入一定时间段所能处理的商品数量，精确到两位小数。
    • 单位时间（小时）：维护用来计算可变效率的特定时间段长度，单位为小时，精确到两位小数，可变效率即为单位时间内处理数量与单位时间的比值。

# 2.7 装卸效率（计算逻辑与案例）

# 2.7.1 同一提卸货点只有一条运单

  • 计算公式：
    • 基本计算公式：站点总作业时间=固定处理时间+可变处理时间
    • $$ 可变处理时间={ \begin{array}{rcl} 基本可变处理时间，& & {最小可变处理时间 \leq 基本可变处理时间 \leq 最小可变处理时间}\ 最大可变处理时间，& & {基本可变处理时间>最大可变处理时间}\ 最小可变处理时间，& & {基本可变处理时间<最小可变处理时间}\ \end{array} $$
    • $$ 基本可变处理时间= \frac{运单总量}{可变效率}$$
    • $$ 可变效率= \frac{单位时间内的处理数量}{单位时间（小时）}$$

# 2.7.2 同一提卸货点对应多条运单

  • 同一提卸货点的多条运单将被合并到同一个车次上进行配送。
  • 每个运输地点计算一次固定处理时间。
  • 同一提卸货点的每个运单根据可变效率计算，再根据最大/最小可变处理时间调整各自得到可变处理时间。
  • 以上两个步骤计算出的固定及可变处理时间的和即为站点的总作业时间。
  • 基本公式：
    • 站点总作业时间=站点固定处理时间+运单1可变处理时间+运单2可变处理时间+...

# 2.8 装卸效率（商品代码处理）

商品代码对于运输地点装卸效率计算会产生影响：

  • 基本规则
    • 运单明细及站点可变效率中的商品代码可设置为空或任意一种已维护的商品代码。
    • 站点可变效率的商品代码为空时，无论运单明细上的商品代码如何，均按可变效率计算可变处理时间。
    • 站点可变效率的商品代码设置为某个特定商品大类时，只有运单明细中也维护了相同的商品代码时，这一可变效率才生效，否则视为0。
    • 站点可变效率的计算会被订单上指定的提货时长覆盖。
  • 案例
    • 数据维护情况 - 维护了两种商品大类，商品代码分别为A和B； - 站点的装卸效率设置为50件/小时； - 单明细及站点装卸效率中的商品代码可设置为空、A或B；

# 2.9 站点车型对应（字段说明）

运输地点车型对应页字段说明：

  • 车型对应列表信息
    • 类型：维护运输地点车型对应关系类型，当前默认为“可用”。如果车型对应列表为空，则默认该运输地点允许所有车型进入；如果车型对应列表中维护了若干条“可用”类型的车型，则只有这些已维护的车型允许进入该运输地点，而未维护的车型在该运输地点是禁用的
    • 车型代码：设置该车型对应信息的具体车型，可以下拉搜索选择车型页面中已维护的车型代码，也可以手工输入（存在逻辑校验，只有输入已维护的车型代码才能正常保存）

# 3. 车型维护

# 3.1 基础信息

用户可以通过页面手工维护车型基本信息：

  • 可手动输入配置最大装载重量（必填）、最大装载容积（必填）、最大运单数、最大件数、最大行驶里程数，最大装车长度等信息，系统会根据这些硬约束进行排程；也可通过上传请求批量上传多条车型数据。
  • 可下拉选择车辆类型，如“常温车”、“冷藏车”等，此类型可由用户自行配置。
  • 可下拉选择车型组类型，如“轿车组”、“轻型货车组”、“微型货车组”等，从而系统可根据车型提取相应的高德货车距离矩阵数据用以做路选规划，目前支持“微、轻、中、重”4个等级的货车设置。
  • 可控制该车型是否启用。
  • 车型代码不允许重复。

# 3.2 基础信息（字段说明）

车型基础信息页字段说明：

  • 通用信息：
    • 车型代码：系统车型的唯一代码。
    • 车型描述：维护车型的详细描述，可自由输入相关文本信息。
    • 类别：对车型类别进行分类，可选择“常温车”/“冷藏车”等类型，车型类型字段可在数据字典或后台根据实际需求进行维护或调整。
    • 启用/停用：标明此车型是否启用，停用的车型不会纳入系统优化的范围。
    • 限行组类型：限定车辆所属的限行组类型，系统会根据特定的类型提取相应的距离矩阵数据进行优化。可选择的限行组包括轿车组/微型货车组/轻型货车组等类别，同样可以根据需要进行调整。
    • 最大装载重量（千克）：设定车型的载重上限，单位为千克，精确到小数点后两位数。系统在派单时会保证车型的总载重严格控制在该上限范围内。
    • 最大装载容积（立方米）：设定车型的装载容积上限，单位为立方米，精确到小数点后两位数。系统在派单时会保证车型的总容积严格控制在该上限范围内。
    • 最大装车长度（米）：设定车型能容纳的单个商品的最大长度，通常和车型内径长度一致。系统在计算时会考虑运单上的装车长度来判断是否能将运单放入相应车型，计算时已装车长度不随着运单增加而累加。
    • 最大运单数：设定车型所能装载的运单数量上限。
    • 最大件数：设定车型所能装载的运单商品数量上限。
    • 最大行驶里程数（千米）：设定车型所能行驶的里程上限，系统会保证每个车次不超过特定路线长度。
    • 所属公司/部门：设置车型的所属部门和公司信息，可下拉选择，新增时若未选择则默认为当前账号的所属公司及部门信息。
  • 尺寸与装载信息：
    • 此部分信息仅做记录使用，不涉及算法约束。

    • 若某车型只能装运某种商品：
    • 若某车型不能装运某种商品：
    • 若某车型上两种商品存在互斥关系，即装了一种之后不能继续装运另一种商品：
    • 优化时，系统根据上述车型与商品对应关系，安排对应的车型进行运输。

  • 车型商品对应页字段说明（商品对应列表信息）：

    • 类型：维护车型商品对应关系类型，共有“互斥”（EXCLUDE）/“只能装运”（RESTRICT-IN）/“不能装运”（RESTRICT-OUT）三种类型可供选择，分别对应了车型上两种商品不能同时装运/车型只能装运某些商品/车型不能装运某些商品的场景，可根据实际情况进行选择。
    • 商品代码：设置该商品对应关系所涉及的商品代码；“只能装运”/“不能装运”只需要维护一种商品代码，而“互斥”类型需要维护两种商品；商品代码可以通过下拉搜索选择，也可以手工录入（但存在逻辑校验，只有输入预先维护的商品代码才能正常保存）。

# 4. 车辆维护

# 4.1 基础信息

维护车辆信息时的规则说明：

  • 每个车辆必须挂靠在唯一一种车型下；
  • 可输入车辆的车牌号信息，选择对应的车型将会自动获取体积/重量等属性；
  • 车牌号不允许重复，可针对不同车辆设置不同的体积/重量属性。如果另行设置，则车辆上的属性参数值将覆盖所属车型的参数值，供算法进行计算；反之，如果未另行设置，则取该车辆所属的车型属性值计算；
  • 车牌号必须输入真实车牌的格式，主要分为普通民用车牌与新能源车牌，前者基本格式为“省份简称(1位汉字) + 城市编码(1位字母) + 字母和数字的组合序号(5位)”，后者基本格式为“省份简称(1位汉字) + 城市编码(1位字母) + 字母和数字的组合序号(6位)”，并且序号不能出现字母I和字母O，字母须大写。填入其他格式的车牌号不能正常保存。

# 4.2 基础信息（字段说明）

车辆的基础信息字段说明如下：

  • 车辆ID：系统自动生成的唯一序列号，不可修改。
  • 车牌号：车辆车牌号，保存后不可修改。
  • 车型代码：表明车辆所属的车型。选择后会自动带出车型的体积重量等约束。
  • 体积重量件数运单数等约束：默认与车型上的配置相同，可以单独修改车辆上的维度。
  • 启用/停用：标明此车辆是否启用，停用的车辆不会纳入系统优化的范围。
  • 所属公司/部门：下拉选择，设置车辆的所属公司和部门信息。

# 5. 承运商维护

# 5.1 基础信息

维护承运商基本信息时的规则说明：

  • 系统可管理承运商代码、简称、描述、类型、启停用、联系方式等信息。
  • 承运商代码不允许重复。

# 5.2 基础信息（字段说明）

承运商页字段说明如下：

  • 通用信息
    • 承运商代码：系统中承运商的唯一代码。
    • 承运商简称：设置该承运商的简称，方便辨别。
    • 承运商描述：可通过文本录入的方式增加承运商的详细描述。
    • 类型：可设置承运商的类型，当前默认选项为承运商（CARRIER）/装卸商（HANDLER），可在数据字典页面中查看类型信息，也可根据需要在后台进行调整。
    • 启用/停用：标明此承运商是否启用，停用的承运商及其车辆资源不会纳入系统优化的范围。
    • 电话：可维护承运商的电话信息。
    • 邮箱：可维护承运商的邮箱信息。
    • 所属公司/部门：设置承运商的所属公司和部门信息，新增时若未选择则默认为当前账号的所属公司及部门信息。

# 6. 承运商资源维护

# 6.1 基础信息

承运商资源指该承运商可用的车型、车辆与他们的成本与数量等，维护承运商基本信息时的规则说明：

  • 可查询录入承运商与车型信息，建立对应关系，从而维护承运商可用车型数量、调度成本、启/停用等信息。
  • 系统优化的时候遵循此处设置的车型数量上限，若不开启计费引擎，则调度成本最低会作为优化目标。
  • 维护承运商可用资源数据时，承运商代码、车型与车型代码不允许同时重复。
  • 在指定承运商、车型代码后，可以在明细中查询添加承运商车辆信息。
  • 车型数量要求不小于添加的明细车辆数。
  • 针对承运商某车型，可指定对应的出发/返回车场，以应对回仓需求。

# 6.2 基础信息(字段说明)

承运商可用资源页面的字段说明如下：

  • 通用信息：
    • ID：承运商资源表唯一主键，不可编辑，由系统自动生成。
    • 承运商代码：可下拉选择或手工录入已维护的承运商代码。
    • 承运商简称：根据选择的承运商由系统自动填充显示，不可编辑。
    • 车型代码：可下拉选择或手工录入已维护的车型代码；承运商代码和车型代码共同构成承运商资源的唯一代码，即承运商资源页中承运商代码和车型代码不能同时相同。
    • 车型描述：根据选择的车型由系统自动填充显示，不可编辑。
    • 车型数量：维护承运商可用的车型数量上限，系统会根据这一限制严格控制所能使用的车型数量。
    • 车场：表明该资源下的车辆的出发车场以及完成配送任务后的返回车场。
    • 车型调度成本（元）：维护承运商使用车型的调度成本，单位为元，精确到两位小数，即整车价；系统会根据这一成本信息作为重要输入权衡不同的车辆分配方案从而寻找最优解，开启计费引擎则以计费引擎为准。
    • 距离调度成本（元）：维护承运商使用车型的距离成本，单位为元/公里，精确到两位小数；即该车型按照行驶距离计算的每公里调度成本；系统会根据这一成本信息作为重要输入权衡不同的距离、车辆分配方案从而寻找最优解，开启计费引擎则以计费引擎为准。
    • 启用/停用：标明此承运商资源是否启用，停用的承运商资源不会纳入系统优化的范围。
    • 所属公司/部门：下拉选择，设置承运商资源的所属公司和部门信息，必须精确到部门。
  • 车辆信息：
    • 填入对应车辆信息，表明该承运商有多少辆具体的车型。

# 7. 商品类别维护

# 7.1 基础信息

维护商品类别时的规则说明：

  • 可输入商品代码及描述，并通过下拉框调整启/停用状态。
  • 商品代码不允许重复。

# 7.1 基础信息 （字段说明）

  • 通用信息：
    • 商品代码：系统中商品类别的唯一代码
    • 商品描述：可通过文本录入的方式增加维护商品类别的详细描述
    • 启用/停用：标明此商品类别是否启用，停用的商品类别不会纳入系统优化的范围
    • 所属公司/部门：下拉选择，设置商品类别的所属公司和部门信息；如只维护公司不维护部门信息，则该商品类别在该公司下所有部门可用；如未维护公司和部门信息，则该商品类别隶属于当前帐号的所属公司/部门
